\begin{abstract}
Temeraire is a hugepage-aware extension of TCMalloc that aims to improve
application performance by increasing hugepage coverage while avoiding excessive
fragmentation. The original OSDI 2021 paper evaluates Temeraire in individual
applications, a Google fleet experiment, and a production rollout. This report
explains the underlying memory-management problem, summarizes the design of
Temeraire, and focuses on the part that is realistically reproducible with
public code: the Redis 6.0.9 case study. A Docker-based artifact was built to
compare a historical public TCMalloc revision in two configurations: a
legacy-pageheap configuration and a hugepage-aware configuration. The artifact
records software versions, Transparent Huge Page settings, benchmark
parameters, allocator preload state, and raw Redis benchmark outputs. The
resulting reproduction is methodologically useful but intentionally narrow: it
can test whether the Redis benchmark shape can be recreated, but it cannot
reproduce Google's warehouse-scale conditions or fleet-level claims.
\end{abstract}

\keywords{memory allocation, huge pages, TCMalloc, Temeraire, Redis,
reproducibility}

\section{Introduction}

Memory allocation is not only a local implementation detail of C and C++
programs. In large systems it can influence memory placement, hugepage
availability, and therefore the number of expensive address translation events.
Hunter et al. argue that allocator design can improve ``fleet efficiency'' by
reducing overhead outside the allocator itself \cite{273711}. Their allocator
extension, Temeraire, changes the pageheap of TCMalloc so that allocations are
placed with awareness of hugepage-sized regions.

The central idea is that an allocator should not be evaluated only by the time
spent inside \texttt{malloc} and \texttt{free}. Allocation decisions determine
where objects are placed in virtual memory. This placement affects whether the
operating system can use huge pages, and huge pages affect the number of
translation lookaside buffer (TLB) misses in later application execution. A
slightly more expensive allocator decision can therefore be profitable if it
reduces stalls in the application code that uses the allocated memory.

The goal of this seminar artifact is not to repeat the complete paper. The
paper's most important results come from Google's production environment, where
many workloads share warehouse-scale computers and where telemetry is available
across large machine populations. That environment is not public. A faithful
seminar reproduction must therefore restrict its claim. The reproducible target
chosen here is the Redis case study described in the paper: Redis 6.0.9 is run
with TCMalloc's legacy pageheap and with a hugepage-aware configuration, and
both configurations are measured with repeated \texttt{redis-benchmark} trials.

This report has three contributions. First, it explains the technical
background needed to understand why a hugepage-aware allocator can matter.
Second, it summarizes the design of Temeraire and its main placement heuristic.
Third, it documents a public reproduction artifact whose main design goal is
reproducibility: pinned source versions, containerized setup, explicit benchmark
parameters, raw output preservation, and documented deviations from the paper.

\section{Background}

\subsection{Paging and Address Translation}

Modern operating systems provide each process with a virtual address space.
Program instructions therefore operate on virtual addresses rather than directly
on physical RAM addresses. The processor and operating system translate virtual
pages to physical page frames through page tables. Since page-table walks are
expensive, processors cache recent translations in translation lookaside buffers
(TLBs).

With ordinary 4 KiB pages, a large working set can require many translations.
For example, a 2 MiB memory region consists of 512 ordinary 4 KiB pages. If the
working set exceeds the effective reach of the TLB, the processor must perform
more page-table walks. These walks can stall execution and reduce application
throughput.

\subsection{Huge Pages}

Huge pages reduce address translation overhead by mapping larger virtual memory
regions with fewer page-table entries. On x86 systems, a common huge page size
is 2 MiB. This means that one TLB entry can cover the same virtual memory range
that would otherwise require 512 ordinary 4 KiB page mappings. The benefit is
not that a whole 2 MiB region is loaded into a CPU cache at once. Rather, the
benefit is that fewer cached address translations are needed to cover the same
amount of virtual memory.

The performance benefit comes with constraints. A hugepage mapping is useful
only when the virtual range is aligned and can be backed by a suitable physical
memory range. Linux Transparent Huge Pages can form such mappings
automatically, but the kernel's ability to do so depends on the memory layout
exposed by the application and its allocator. If an allocator scatters live
objects across many hugepage-sized regions, the operating system may be unable
to keep those regions as intact huge pages.

\subsection{Fragmentation Trade-off}

The central difficulty is the trade-off between memory efficiency and TLB
efficiency. Returning unused memory in small pieces can reduce resident memory
usage, but it may require the kernel to split a hugepage mapping into ordinary
pages. Keeping the hugepage intact preserves TLB coverage, but can waste memory
if only a small part of the 2 MiB region still contains live data.

Figure \ref{fig:fragmentation} illustrates the basic problem. Free memory exists
inside the region, but it is scattered between live allocations. An allocator
that returns the free pieces to the operating system may break hugepage
coverage. An allocator that keeps the whole region backed may retain physical
memory that is currently not useful to the application.

A hugepage-aware allocator therefore has two goals that appear to be in
tension. It should keep frequently used memory on huge pages to reduce TLB
misses, but it should also avoid retaining large amounts of unused physical
memory. Temeraire addresses this by changing allocation placement decisions
inside TCMalloc's pageheap.

\section{Temeraire}

\subsection{TCMalloc Context}

TCMalloc is structured as a hierarchy of allocation caches. Small allocations
are usually served by local and central caches. Larger page-granularity requests
eventually reach the pageheap, which obtains memory from the operating system
and returns unused memory when possible. Temeraire changes this backend part of
TCMalloc. It does not replace the entire allocator; instead, it provides a
hugepage-aware pageheap implementation.

Figure \ref{fig:tcmalloc} shows the relevant context. TCMalloc serves many
requests from CPU-local and central caches. Only a smaller fraction of requests
reaches the pageheap. This makes Temeraire's design unusual: the pageheap may
spend more time making placement decisions, because these decisions affect the
later execution of application code through hugepage coverage.

This design choice is important because pageheap operations are not necessarily
the dominant direct cost of allocation. The paper's argument is broader:
slightly more expensive pageheap decisions can be worthwhile if they improve
the placement of objects and thereby reduce address translation costs in the
application.

The \HugeAllocatorLabel{} manages hugepage-sized virtual address ranges. It can
provide aligned memory ranges to the other components. The \HugeCacheLabel{}
stores fully empty, backed hugepages so that short-term allocation fluctuations
do not immediately force expensive release and re-backing operations.

The \HugeFillerLabel{} is the core component for small allocations. It
packs allocations into partially used hugepages. Its goal is not merely to find
any free space, but to choose placements that preserve useful free ranges and
increase the chance that some hugepages become completely empty.

The \HugeRegionLabel{} handles allocations that are too large for the filler
but too small to round up without producing excessive slack. For example, many
allocations slightly larger than 2 MiB would waste almost a full hugepage each
if each request were rounded independently. A region allows such allocations to
cross hugepage boundaries inside a larger range.

Finally, the \textit{HugePageAwareAllocator} routes requests between these
components. Small allocations are directed toward the filler. Larger allocations
may use regions or complete hugepage ranges. This size-based routing is a
central part of the design because different allocation sizes create different
fragmentation risks.

\subsection{Placement Heuristic}

The most important implementation idea is the policy used by the
HugePageFiller. A simple best-fit strategy is not sufficient because the
allocator does not only care about the total amount of free space. It also cares
about the shape of that free space. A hugepage with one long contiguous free
range is more valuable than a hugepage with many small holes, because the long
range can still satisfy larger future requests.

Temeraire therefore prioritizes hugepages by their longest free range. For a
request of size \(K\), the allocator prefers hugepages whose longest free range
is just large enough to satisfy the request. This preserves longer free ranges
for future large requests. Within groups of equal longest-free-range status, the
allocator prefers fuller hugepages, using the number of allocations as a
heuristic. The intuition is that a hugepage containing fewer live allocations is
more likely to become completely empty and should therefore be disturbed less.

This heuristic expresses the paper's main design philosophy. The allocator is
allowed to spend more effort making placement decisions because a bad placement
can create long-lived fragmentation, while a good placement can improve
hugepage coverage and reduce TLB stalls in application code.

\section{Related Work}

Prior work on memory allocation and huge pages can be grouped into
allocator-side approaches and operating-system-side approaches. Temeraire is
closest to allocator-side work because it changes object placement in user
space rather than relying only on the kernel to recover hugepage-friendly
layouts after the fact.

SuperMalloc is a scalable multithreaded allocator for 64-bit systems
\parencite{10.1145/2754169.2754178}. It is primarily designed for allocation
speed and scalability, but it also considers huge pages for very large
allocations. Compared with Temeraire, this is a more limited use of huge pages:
SuperMalloc does not attempt to make the general pageheap layout
hugepage-aware for diverse allocation sizes. Temeraire instead targets the
fragmentation problem that arises when many smaller and medium-sized
allocations interact with Transparent Huge Pages.

MallocPool also addresses TLB efficiency, but with a different abstraction
\parencite{8502619}. Its approach is to preallocate a large
memory region and allocate chunks from it contiguously, so that dynamically
allocated objects can benefit from more favorable TLB mapping. This is related
to Temeraire's goal of improving translation behavior through allocation
layout. However, Temeraire is integrated into TCMalloc's general-purpose
pageheap and must handle a wider range of allocation sizes, release decisions,
and production server workloads.

LLAMA takes another allocator-side approach. It uses learning-based lifetime
prediction for C++ server workloads and organizes allocations by predicted
lifetime classes \parencite{10.1145/3373376.3378525}. The motivation is that huge pages
are most effective when objects with compatible lifetimes are placed together:
if short-lived and long-lived objects are mixed on the same hugepage, the
long-lived objects can prevent the whole hugepage from becoming empty.
Temeraire does not require machine-learning predictions. Instead, it uses
online placement heuristics, especially longest-free-range information and
allocation counts, to reduce fragmentation and improve hugepage coverage.

Operating-system approaches such as Ingens and HawkEye manage huge pages from
the kernel side \parencite{10.5555/3026877.3026931,10.1145/3297858.3304064}. These systems
try to allocate, distribute, or reclaim huge pages more effectively without
changing the application allocator itself. Such approaches are complementary to
Temeraire but address a different layer of the problem. A kernel can only work
with the virtual memory layout exposed by user space. If the allocator scatters
live objects across hugepage-sized regions, kernel-level hugepage management
alone may still face fragmentation and memory bloat.

Overall, prior work shows that huge pages can improve performance but also
introduce fragmentation and memory-efficiency problems. Temeraire's specific
contribution is to treat hugepage coverage as a first-class objective inside a
production allocator backend. It tries to combine the performance benefit of
huge pages with controlled memory overhead by packing some hugepages densely
and making others empty enough to return to the operating system.

\section{Original Evaluation Methodology}

The original paper evaluates Temeraire at several levels. First, it studies
individual production applications and reports application-level metrics such as
throughput, latency, memory usage, instructions per cycle, TLB-related stalls,
and time spent in allocation. Second, it includes a Redis case study that is
more accessible to external reproduction because Redis is open source. Third,
the paper uses a fleet experiment in which Temeraire is enabled on a fraction
of machines and compared with a control group. Finally, the authors report
longitudinal data from the production rollout.

This layered methodology is important for interpreting the claims. The Redis
case study can show whether the allocator change affects a concrete public
application. The fleet experiment and rollout data address a different question:
whether hugepage-aware allocation improves aggregate efficiency across diverse
warehouse-scale workloads. The seminar artifact reproduces only the Redis-style
part of this evaluation, not the production fleet methodology.

The paper reports that the Redis experiment used Redis 6.0.9 and compared
Temeraire with TCMalloc's legacy pageheap. Each configuration was evaluated
with repeated \texttt{redis-benchmark} trials. In each trial, the benchmark
issued 1,000,000 requests that pushed five elements and read those five
elements. This part of the evaluation is a suitable target for a seminar
artifact because the application and benchmark are publicly available.

\section{Artifact Design}

The repository is organized around a Docker container. The container provides a
Linux environment with compiler tools, Redis tooling, \texttt{perf}, and build
dependencies. The host directory is mounted into the container so that raw
outputs are preserved in the repository tree.

The setup script pins the relevant public source versions:

\begin{itemize}
  \item Redis \texttt{6.0.9};
  \item \texttt{gperftools-2.16}, used only as an optional side baseline;
  \item public \texttt{google/tcmalloc} commit
  \texttt{8e534f507074}.
\end{itemize}

The historical TCMalloc revision is used because it still exposes the
\texttt{want\_no\_hpaa} mechanism needed to construct a legacy pageheap variant.
A small wrapper target is added inside the cloned TCMalloc tree. The setup then
builds two shared objects: \\ \texttt{libtcmalloc\_legacy.so} and
\texttt{libtcmalloc\_temeraire.so}. Benchmark scripts select between them with
\texttt{LD\_PRELOAD}.

This source choice needs to be interpreted carefully. The public TCMalloc
repository is the current allocator codebase and not a frozen paper artifact.
The selected commit is therefore a practical approximation that supports the
legacy-versus-hugepage-aware comparison. It should not be presented as proof
that the exact internal implementation used in the original paper was
reconstructed.

\section{Reproduction Methodology}

The main workflow is:

\begin{lstlisting}[style=commandstyle]
docker compose build
docker compose run --rm temeraire-dev bash -lc "./scripts/setup_env.sh"
docker compose run --rm temeraire-dev bash -lc "./scripts/check_allocator_preload.sh"
docker compose run --rm temeraire-dev bash -lc "./scripts/collect_system_info.sh"
docker compose run --rm temeraire-dev bash -lc "./scripts/run_redis_benchmark.sh legacy"
docker compose run --rm temeraire-dev bash -lc "./scripts/run_redis_benchmark.sh temeraire"
\end{lstlisting}

The benchmark script starts Redis, selects the allocator mode, records memory
state, and executes repeated benchmark trials. Each trial first flushes the
database and then performs two operations:

\begin{itemize}
  \item \texttt{LPUSH benchmark:list v1 v2 v3 v4 v5};
  \item \texttt{LRANGE benchmark:list 0 4}.
\end{itemize}

The default configuration follows the paper-shaped Redis workload: 2000 trials,
1,000,000 requests per trial, 50 clients, and pipeline depth 16. A separate
smoke-test mode can reduce trial and request counts to validate the environment
quickly, but those smoke-test results must not be compared to the paper.

Reproducibility metadata is collected before the benchmark. This includes kernel
version, operating-system release information, CPU model, memory summary,
Transparent Huge Page settings, Redis version, TCMalloc commit, and allocator
artifacts. The allocator preload check is important because Redis may report
\texttt{mem\_allocator: libc} even when the process allocator is actually
changed through \texttt{LD\_PRELOAD}.

The main measured value is throughput in requests per second as reported by
\texttt{redis-benchmark}. The artifact also preserves raw benchmark logs so that
variance across trials can be inspected. If available on the host system,

\section{Discussion}

The strongest part of this artifact is reproducibility of procedure. It records
the exact public source versions, benchmark parameters, allocator mode, raw
outputs, and system context. This makes the experiment inspectable and easier to
rerun on another machine.

The main limitation is fidelity. The original Redis experiments were run on
Intel Skylake Xeon servers with a specific LLVM revision and Google's internal
TCMalloc environment. A Docker container on a different host cannot guarantee
the same kernel behavior, THP state, physical memory fragmentation, NUMA
topology, or performance counter behavior. These factors matter especially for
huge pages because hugepage availability is a property of the whole system, not
only of the benchmark process.

Another limitation is source identity. The public TCMalloc commit used here is
a historical approximation chosen because it permits a legacy-versus-Temeraire
split. It should not be presented as proof that the exact internal paper
artifact was reconstructed. This is especially important because the current
TCMalloc repository is broader than the OSDI 2021 Temeraire design and has
continued to evolve.

Despite these limitations, the reproduction remains useful. It forces the
paper's Redis methodology into explicit, runnable steps and gives a concrete way
to discuss which parts of systems papers are reproducible from public artifacts
and which parts depend on inaccessible production context. It also makes the
central technical claim testable at small scale: allocator placement decisions
can affect application behavior beyond the direct cost of allocation itself.

\section{Conclusion}

This artifact provides a narrow but locally reproducible reproduction path for
the Temeraire paper. Instead of claiming to reproduce fleet-scale results in
WSCs, it focuses on the Redis case study and documents all relevant setup
choices. The expected final outcome is not a universal confirmation of the
paper's production claims, but a well-scoped comparison of legacy TCMalloc and
hugepage-aware TCMalloc behavior under a public, repeatable Redis benchmark.

The broader lesson of the paper is that memory allocation can influence system
performance through memory layout, not only through the direct speed of
\texttt{malloc} and \texttt{free}. Huge pages can reduce TLB pressure, but they
also create fragmentation challenges. Temeraire addresses this by packing
allocations in a way that tries to keep some hugepages dense and others empty
enough to release. The seminar reproduction examines this idea in a controlled
public setting while explicitly separating local evidence from Google's
fleet-scale evaluation.
